# Scripts README

## Local Split Dev

Use [`local-split-dev.sh`](/Users/sotheakh/Documents/develop/sv-tms/scripts/local-split-dev.sh) for the current local TMS web stack.

Common usage:

```bash
cp scripts/local-split-dev.env.example scripts/local-split-dev.env
./scripts/local-split-dev.sh start
./scripts/local-split-dev.sh status
./scripts/local-split-dev.sh status backend
./scripts/local-split-dev.sh ps
./scripts/local-split-dev.sh ps --long backend
./scripts/local-split-dev.sh env driver
./scripts/local-split-dev.sh smoke
./scripts/local-split-dev.sh logs auth
./scripts/local-split-dev.sh logs -f backend
SKIP_BUILD=1 ./scripts/local-split-dev.sh restart backend
./scripts/local-split-dev.sh stop frontend
./scripts/local-split-dev.sh clean
```

What it manages:
- `tms-backend` on `:8080`
- MySQL, Redis, Mongo via `docker-compose.local-dev.yml`
- `tms-auth-api` on `:8083`
- `tms-driver-app-api` on `:8084`
- `tms-frontend` on `:4200`

Notes:
- Local JWT defaults are built into shared config, so the launcher works without extra env setup.
- `SKIP_BUILD=1` skips the pre-start Maven install step and uses existing `target/*.jar` artifacts for backend/auth/driver when available.
- `status backend|auth|driver|frontend` gives a single-service check.
- `ps` shows pid, port, launch mode, and a compact live process summary for managed services.
- `ps --long` prints the full raw process command when you need exact launch details.
- `env backend|auth|driver|frontend` prints the effective launcher inputs and whether a service will use `jar`, `maven`, or `npm` mode.
- `restart backend|auth|driver|frontend` lets you bounce one service without touching the rest of the stack.
- `logs -f backend|auth|driver|frontend` follows a single managed log file.

This folder contains helper scripts used for transforming legacy SQL dumps and producing developer-friendly, idempotent upserts.

Key scripts
- `make_dev_compatible_vehicles_sql.py` — conservative transformer that reads `merge_legacy_vehicles_per_row.sql` and writes `merge_legacy_vehicles_per_row_dev_compat.sql`. It removes the legacy `year` column and the corresponding `year`=VALUES(...) assignments. Use this script when the legacy dump contains `year` but the dev schema uses `year_made`.

- `cleanup_dev_vehicles_sql.py` — post-processing normalization. It removes stray `year` tokens, collapses duplicate commas, fixes double semicolons and ensures each statement ends with `;\n`. It also removes orphaned `ON DUPLICATE KEY UPDATE` lines introduced by earlier transforms.

- `merge_legacy_vehicles_per_row.sql` — source file containing per-row INSERT...ON DUPLICATE statements (legacy output). Do not edit this file directly unless you understand the incoming dump format.

Common usage
```bash
# generate dev compatible file
python3 scripts/make_dev_compatible_vehicles_sql.py

# run cleanup
python3 scripts/cleanup_dev_vehicles_sql.py

# verify no legacy tokens left
grep -n "\byear\b" scripts/merge_legacy_vehicles_per_row_dev_compat.sql || true

# optional dry-run: import into a temporary MySQL container
docker run --name temp-mysql -e MYSQL_ALLOW_EMPTY_PASSWORD=yes -d mysql:8
sleep 10
docker exec -i temp-mysql mysql -uroot < scripts/merge_legacy_vehicles_per_row_dev_compat.sql || true
docker rm -f temp-mysql
```

Troubleshooting
- If `grep` still finds `year` tokens, inspect `merge_legacy_vehicles_per_row_dev_compat.sql` for malformed ON DUPLICATE fragments. The cleanup script attempts to remove stray fragments; if an edge-case remains, open an issue and include the problematic snippet.
- If the MySQL import fails with "Unknown column 'year'" — that means a `year` reference remains. Re-run the transformer and cleanup, then re-check.

Notes
- These scripts are intentionally conservative: they only edit lines that match `INSERT INTO `vehicles`` patterns and will fall back to leaving a line unmodified if parsing fails. That helps avoid accidental corruption. Always validate output before applying.
Scripts for migration helpers

generate_per_row_upserts.py
  - Purpose: Split mysqldump multi-row INSERT ... VALUES(...),(...); statements into
    single-row INSERT ... VALUES (...) ON DUPLICATE KEY UPDATE ...; statements.
  - Usage:
      python3 generate_per_row_upserts.py <input.sql> <output.sql>

Recommended workflow
  1. Create a full backup of the target database (dev/staging) before applying any migration.
  - You can use `scripts/backup_dev_full.sh` to create a utf8mb4-preserving dump from a running mysql container.
  2. Run this script on your mysqldump generated data-only files (drivers, vehicles, etc.).
     Example:
       python3 generate_per_row_upserts.py scripts/merge_legacy_vehicles_dev_compatible.sql scripts/merge_legacy_vehicles_per_row.sql
  3. Manually review the generated <*_per_row.sql> for columns/constraints that may not exist in the target schema.
  4. Run the resulting SQL against a restored staging environment first. Verify counts and referential integrity.
  5. If staging is good, take a fresh dev backup and apply to dev. Avoid using FOREIGN_KEY_CHECKS=0 unless you understand the integrity implications.

Notes
  - The script tries to correctly handle quoted strings and escaped characters, but please inspect the output.
  - Do not run generated SQL on production until you have a tested rollback and approval.
  - For producing dumps, always use mysqldump with --default-character-set=utf8mb4 and import with the same charset to avoid double-encoding.
